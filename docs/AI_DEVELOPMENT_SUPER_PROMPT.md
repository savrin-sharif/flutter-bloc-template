```text
lib/
├── app/                         # Root application
├── core/
│   ├── config/                  # Environment selection
│   ├── di/                      # get_it composition root
│   ├── errors/                  # Shared failure types
│   ├── network/                 # Dio and connectivity infrastructure
│   ├── presentation/mixins/     # Shared BLoC behavior
│   ├── router/                  # go_router graph

    │   ├── datasources/         # Dio/local persistence calls
    │   ├── models/              # JSON DTOs and mapping
    │   └── repositories/        # Domain contract implementations
    ├── domain/
    │   ├── entities/            # Framework-independent business objects
    │   ├── repositories/        # Abstract contracts
    │   └── usecases/            # Focused application operations
    └── presentation/
        ├── bloc/                # Events, immutable states, BLoCs
        ├── pages/               # Routed screens
```

Dependencies point inward:

```text
presentation ──> domain <── data
       │                    │
       └──── core APIs ─────┘
```

- Domain must not import Flutter, Dio, JSON, widgets, or data-layer classes.
- Data implements domain repository contracts.
- Presentation calls use cases, not Dio or data sources directly.
- Register concrete implementations only in `core/di/injection.dart`.
- Shared widgets are added only for stable, genuinely cross-feature behavior.
  Prefer feature-local widgets over speculative global abstractions.

## REST API rules

The existing singleton `Dio` is the only REST client. It already uses
`AppConfig.baseUrl`, timeouts, JSON accept headers, and debug logging.

Environment behavior is intentional:

- Release builds always use `PROD_URL`.
- Debug/profile builds use `DEV_URL` by default.
- Debug/profile can use `LOCAL_URL` with
  `--dart-define=APP_ENV=local`.
- Never hardcode a host inside a feature data source.
- Endpoint paths may be constants local to a feature or a focused core endpoint
  definition; do not mix hosts with paths.

Before implementing an API, obtain or discover these details:

- Endpoint paths and HTTP methods
- Request body/query/path parameters
- Success status codes
- Exact response envelope and JSON fields
- Pagination shape, if any
- Authentication mechanism and token source
- Validation/error response shapes
- Whether delete returns content, `204`, or a soft-deleted object

If these details are not in the request, repository, API documentation, or
provided examples, ask concise questions. Do not invent production schemas.
Reasonable UI-only assumptions may be stated explicitly, but API-contract
assumptions require confirmation.

### Remote data source

Create an abstract contract and Dio implementation, for example:

```dart
abstract interface class ItemRemoteDataSource {
  Future<List<ItemModel>> getItems({int? page, int? limit});
  Future<ItemModel> getItem(String id);
  Future<ItemModel> createItem(CreateItemModel input);
  Future<ItemModel> updateItem(String id, UpdateItemModel input);
  Future<void> deleteItem(String id);
}

final class ItemRemoteDataSourceImpl implements ItemRemoteDataSource {
  const ItemRemoteDataSourceImpl(this._dio);

  final Dio _dio;

  @override
  Future<ItemModel> getItem(String id) async {
    final response = await _dio.get<Map<String, Object?>>('/items/$id');
    final json = response.data;
    if (json == null) {
      throw const FormatException('The server returned an empty item.');
    }
    return ItemModel.fromJson(json);
  }
}
```

Adapt parsing to the real response envelope. Validate nullable/optional values
at the boundary. Avoid unchecked `dynamic` chains and unsafe casts. A malformed
successful response must fail predictably instead of silently creating invalid
domain data.

### Models and entities

- Domain entities express business meaning and equality.
- Data models own `fromJson`, `toJson`, wire names, and mapping.
- Do not put `Dio`, `Response`, JSON maps, or transport-only fields in entities.
- Separate create/update input types when their required fields differ.
- Do not send server-managed fields such as IDs or timestamps unless required.
- Handwritten mapping is acceptable. Add code generation only when the project
  already uses it or the feature benefits enough to justify the dependency.

Example boundary:

```dart
final class ItemModel {
  const ItemModel({required this.id, required this.name});

  factory ItemModel.fromJson(Map<String, Object?> json) {
    final id = json['id'];
    final name = json['name'];
    if (id is! String || name is! String) {
      throw const FormatException('Invalid item response.');
    }
    return ItemModel(id: id, name: name);
  }

  final String id;
  final String name;

  Item toEntity() => Item(id: id, name: name);
}
```

### Repository and failure behavior

Define the repository interface in domain and implement it in data. Repository
methods return domain entities, never DTOs or Dio responses.

Use the project’s existing failure policy consistently. If the feature’s API
needs richer failures, extend `core/errors/failure.dart` deliberately with
useful categories such as validation, unauthorized, not found, conflict,
connection, and server failures. Do not parse error strings in widgets.

The provided `AsyncControllerMixin` already:

- Tracks named operations independently
- Emits immutable loading snapshots
- Guarantees operation cleanup
- Extracts common Dio error fields (`message`, `detail`, `error`, `errors`)
- Falls back to a safe user-facing error
- Classifies likely connection failures

Use it in feature BLoCs rather than duplicating `try/catch/finally` around every
event. API-specific status handling may be added at the data/repository boundary
when the generic behavior is insufficient.

## CRUD BLoC design

Use explicit events. A typical CRUD feature may include:

```text
ItemsRequested
ItemRefreshRequested
ItemCreateSubmitted
ItemUpdateSubmitted
ItemDeleteRequested
```

Choose operation keys that allow independent progress indicators:

```text
items.load
items.refresh
items.create
items.update.<id>
items.delete.<id>
```

State should be immutable and `Equatable`. Preserve existing successful data
while a mutation runs. Do not replace an entire populated page with a global
spinner for one row’s delete operation.

A useful state normally contains:

- Load/status state
- Current immutable entity collection or detail
- `activeOperations`
- User-safe error information
- Field validation errors when returned by the API
- Pagination metadata when applicable

Use `AsyncControllerMixin<Event, State>`:

```dart
class ItemsBloc extends Bloc<ItemsEvent, ItemsState>
    with AsyncControllerMixin<ItemsEvent, ItemsState> {
  ItemsBloc(this._getItems) : super(const ItemsState()) {
    on<ItemsRequested>(_onRequested);
  }

  final GetItems _getItems;

  Future<void> _onRequested(
    ItemsRequested event,
    Emitter<ItemsState> emit,
  ) => runControllerTask<List<Item>>(
    key: 'items.load',
    emit: emit,
    task: _getItems.call,
    onLoadingChanged: (current, operations) => current.copyWith(
      activeOperations: operations,
    ),
    onSuccess: (current, items) => current.copyWith(
      status: ItemsStatus.success,
      items: List.unmodifiable(items),
      errorMessage: null,
    ),
    onFailure: (current, error) => current.copyWith(
      status: ItemsStatus.failure,
      errorMessage: error.message,
      isConnectionError: error.isConnectionError,
    ),
  );
}
```

Ensure `copyWith` can intentionally clear nullable errors. A sentinel-based
`copyWith`, explicit clear flag, or equivalent is safer than an implementation
where `null` ambiguously means both “retain” and “clear.” Collections stored in
state must not expose mutable references.

For create/update/delete success, update local state deterministically:

- Create: insert or append the server-returned entity according to sort order.
- Update: replace the matching ID with the server-returned entity.
- Delete: remove the matching ID only after confirmed success, unless an
  explicitly requested optimistic workflow includes rollback.
- Refresh: normally retain current content while showing refresh progress.
- Pagination: deduplicate by stable ID and prevent overlapping page requests.

Choose an event transformer deliberately if repeated events can race. Use
restartable behavior for search/type-ahead, droppable behavior for duplicate
page loads/submissions, and sequential behavior for ordered mutations. Do not
add a transformer package unless needed; use the APIs already available first.

## UI rules

- BLoCs do not own `BuildContext`, `TextEditingController`, `FocusNode`,
  `GlobalKey<FormState>`, dialogs, snack bars, or navigation.
- Stateful widgets own and dispose text/focus controllers.
- Widgets validate local form syntax; the API/domain validates business rules.
- Use `BlocBuilder`, `BlocSelector`, and `buildWhen` to limit rebuilds.
- Use `BlocListener` for one-time UI reactions and navigation.
- Never show snack bars or dialogs inside a repository, use case, or BLoC.
- Read per-operation loading state so create/update/delete controls disable only
  when their own operation is active.
- Provide loading, empty, populated, recoverable error, offline, and retry UI.
- Prevent duplicate destructive submissions.
- Ask for confirmation before destructive deletion unless product requirements
  explicitly say otherwise.
- Preserve the generated design system. Material pages use Material controls;
  Cupertino pages use Cupertino controls and `CupertinoPageScaffold`.
- When both styles are supported by existing adaptive code, implement both.
- Keep pages focused; extract only meaningful feature widgets.

## Dependency injection and routing

Register dependencies in `configureDependencies()` from the outside inward:

```dart
getIt
  ..registerLazySingleton<ItemRemoteDataSource>(
    () => ItemRemoteDataSourceImpl(getIt<Dio>()),
  )
  ..registerLazySingleton<ItemRepository>(
    () => ItemRepositoryImpl(getIt<ItemRemoteDataSource>()),
  )
  ..registerLazySingleton(() => GetItems(getIt<ItemRepository>()))
  ..registerFactory(() => ItemsBloc(getIt<GetItems>()));
```

- Reuse the registered singleton `Dio`.
- Data sources/repositories/use cases are usually lazy singletons.
- BLoCs are usually factories unless their state intentionally spans routes.
- Dispose resources according to their registered lifecycle.
- Add route constants and `GoRoute`s to the existing `go_router` graph.
- Provide a route-level `BlocProvider` when the BLoC belongs to that route.
- Pass IDs through path parameters and short serializable filters through query
  parameters; fetch authoritative entities by ID rather than placing mutable
  objects in global state.

## Authentication, security, and logging

If authentication is requested:

- Discover the project’s token storage and refresh policy first.
- Add authentication centrally through a Dio interceptor, not separately in
  every feature data source.
- Serialize concurrent refresh attempts and retry a failed request at most once.
- Never log access tokens, refresh tokens, passwords, authorization headers,
  personal information, or secret response bodies.
- Do not place secrets in `.env` assets: Flutter asset values ship with the app.
- Treat client-side authorization checks as UX only; the server remains the
  authority.
